# M11 Design — Eval Efficiency + The Width Re-Gate

**Status:** Design approved 2026-06-11 (brainstorming). Next: implementation plan.

**Thesis:** Make NNUE evaluation cheap per node, then re-ask M10's width question on fair terms. M10 measured 1024 hidden at **−29.91 ± 16.73 (H0)** — not because the bigger net judges worse (training loss improved −4%) but because every accumulator update costs too much and the engine performs too many of them. Shrink eval's share of node time and the same width gets a retrial. Flywheel turn 3 runs in the background, so the milestone banks a data gain regardless of the width verdict: **there is no losing outcome.**

**Context:** 0.10.0 = 3191.9 ± 18.1 anchored (M10: data +87.5 H1 / width −29.9 H0 / buckets ≈ +10 isolated). Rung-d (768+8 buckets, net3d, loss 0.004585) is in its SPRT as this spec is written; its verdict decides the "current-best arch" below. Two flywheel turns have paid +203 anchored on a 592k-param net at ~1.4Mnps AVX2.

---

## 1. Component A — eval efficiency (node-identical refactors)

Two refactors to `src/eval/nnue/accumulator.rs` + `mod.rs`. Both preserve every evaluated number and the exact search tree — they are **infrastructure refactors** under the project's established law: gated by node-identical search at fixed depth + unchanged `Bench:`, committed separately, baseline refreshed afterward, **never credited as Elo**.

### A1 — fused update kernels
Replace the separate per-feature `add`/`sub` full-width loops with fused per-move-shape kernels:
- quiet move: one pass computing `vals[i] += w_add[i] − w_sub[i]` (2 feature columns, 1 traversal)
- capture / en passant: one three-term pass (3 columns, 1 traversal)
- castle: one four-term pass (4 columns, 1 traversal)

Roughly halves accumulator memory traffic. **Arithmetic-order care (a plan-level requirement):** the sequential version's i16 adds never overflow for trained nets; the fused version must keep intermediates in widths/orders that cannot transiently overflow where the sequential version could not (e.g., widen to i32 per element before the combined add, or prove i16-safety from bullet's ±1.98 weight clipping — the plan picks one and tests it).

### A2 — lazy accumulator materialization
- `on_make` no longer touches the accumulator vectors. It resolves the move into its 2–4 feature ops (the same decode logic that exists today: quiet/capture/promo/castle/EP) and pushes them as a **pending delta** entry; cost ≈ a few stores.
- `evaluate()` materializes: applies pending deltas (via A1 kernels) from the last-materialized ply up to the current ply, advancing a watermark.
- `on_unmake` pops; **deltas that were never materialized are discarded free**. TT cutoffs, futility/null prunes, and every other never-evaluated make (~30–40% of makes) stop paying for accumulator updates entirely.
- The `Evaluator` trait API is unchanged (refresh/on_make/on_unmake/evaluate); HCE untouched; search untouched.

### Verification (Component A gates)
1. Full NNUE suite, extended with laziness-specific tests: evaluate-after-N-unevaluated-makes == refresh; make/unmake churn with no evaluate in between; the targeted promo/EP/castle/null set through the lazy path; incremental==refresh random walks (now exercising watermark logic).
2. **Node-identical search** at fixed depth vs the pre-refactor binary (the M5 shared-attack-map precedent) + `Bench:` unchanged.
3. NPS before/after on fixed positions + bench nps recorded in the commit (information, not Elo credit). Expectation: +10–20% NPS at 768; the 1024 tax shrinks proportionally.

## 2. Component B — flywheel turn 3 (background)

~150M positions, identical datagen params (seed 1, 5000 soft nodes, 8 random opening plies, smart-fen-skipping, TB adjudication), **generated by whatever rung-d promotes** (768 plain or 768×8) so the generator is the strongest available. 20 threads (the live bot keeps its cores); starts immediately after the rung-d verdict ships; ~22h. Stats gate / prepare (TMPDIR=tools/data, shuffle mem ≤12288) / byte gate as established.

## 3. Component C — the capstone gates (idle machine, single bot-pause window)

1. **net4a — the data rung:** current-best architecture retrained on turn-3 data; SPRT 8+0.08 [0,10] vs the reigning champion. Likely H1 (turn-2 paid +87); ships within 0.11.0 either way it lands.
2. **net4b — the width re-gate:** 1024 hidden (+8 output buckets if rung-d validated them) trained on the same turn-3 data, SPRT [0,10] vs net4a, **measured on the post-A eval** — the fair retrial. The preserved M10 width patches are re-authored on top of the new accumulator code (small: A2 rewrites accumulator.rs anyway; the width config is consts + struct + asserts).
   - **H1:** width unlocked — ships; M12 opens toward 1536 / king-bucket features on a foundation that can afford them.
   - **H0:** width stays waitlisted with a newly quantified tax; M12 pivots to features-at-768 (king buckets) and/or the search round, with a clean conscience.
3. Winner → anchored gauntlet (300/pairing) vs the **unchanged** M9 pool (comparison row: 0.10.0 = 3191.9) → **ship 0.11.0** → bot redeploy/restart.

## 4. Explicitly out of scope
- **Book audit** (task #39): its own mini-milestone immediately after M11 — independent offline tooling with its own book-on SPRT gate.
- **Repetition-scoring investigation** (task #40): side-quest in any quiet hour; if it surfaces a real fix, it gets its own small gate.
- **Search selectivity round**: M12 candidate, scoped after the width verdict.
- 1536 hidden, king buckets / HalfKA, AVX-512 builds, the MS-A2 bot-VM migration (separate runbook).

## 5. Success criteria
- Component A landed as node-identical commits with measured NPS gain; no Elo credited.
- Turn-3 data banked through all gates; net4a verdict recorded.
- **The width question answered honestly on fair terms** — either outcome is a deliverable.
- 0.11.0 shipped (gauntlet vs unchanged pool), ledgers updated, bot redeployed.

## 6. Risks
- **Lazy-update correctness** is the real engineering risk (watermark/stack bugs are subtle). Mitigations: the strongest test suite in the codebase + the new laziness tests + the node-identical gate, which catches any behavioral drift exactly.
- **Transient i16 overflow** in fused kernels — explicit compute-order analysis + a test on the real net's extreme weights.
- **Width may fail again** — then it fails *priced*, and the roadmap forks to features with evidence.
- **Sequencing**: turn-3 datagen waits on rung-d (hours, not days); the capstone gates need one bot pause (~3–4h including the gauntlet).

## 7. Constraints (unchanged)
std-only runtime; `include_bytes!` embed; self-play-only shipped nets; review-every-step on every commit; SPRT (frozen protocol) then anchored gauntlet; idle-machine measurements; commits to main with the `Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>` trailer; engine-affecting commits carry `Bench:`; `cargo fmt` before committing.
